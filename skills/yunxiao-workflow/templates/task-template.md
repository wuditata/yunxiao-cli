# 任务规范 (Task Template)

本模板适用于具体的研发实现任务。通常由上级需求拆解而来，指引开发或AI助手执行特定代码修改。

## 1. 标题规范与字数限制
> **任务标题规范**：`[父需求摘要] 具体模块的开发或任务`。示例：`[用户画像标签] UserCenter 后端接口开发`
> **子任务标题规范**：`[父级别任务摘要] 具体步骤或动作`。示例：`[UserCenter后端] 新增 getUserTag 接口`
> **字数与完整度要求**：正文控制在 200 字以内。**核心强调**：创建子任务时禁止仅用一句话短语敷衍！即便是子任务，也必须遵照本模板输出完整的实施步骤、前置依赖和自测验证标准 (DoD)。

## 2. 关联需求
> 说明该任务从属于哪个业务需求。
- 需求链接 / 工作项 ID：#XXX

## 3. 具体实施步骤 (How)
尽可能详细地列出实现任务的每一步骤，避免模糊的描述：
1. 修改 `xxx/xxx` 目录下的 `XXX` 模块。
2. 实现 `XXXX` 类中的 `xxx` 接口。
3. 添加对应的数据层映射/查询逻辑。
4. (前端) 调用上文的 `xxx` 接口，绑定/展示到 `XXX` 组件上。
5. （如果涉及数据库）创建或执行 DDL 文件。

## 4. 涉及模块与前置依赖
- **前置依赖**：本任务开始前，有哪些任务必须完成？（例：需要在基础库更新至 1.2.0 版本后开始）
- **影响范围**：修改代码时是否可能影响其他模块（如共用的 utils 方法）？

## 5. 自测 / 验证指南
指明该如何验证此任务是否完成开发：
- [ ] 需包含完整的单测，代码覆盖率达成要求。
- [ ] 若涉及接口，可以通过 `curl` 或 `Postman` 请求并附带返回截图/JSON样例进行确认。
- [ ] 界面改动需自行运行验证功能交互。

## 6. 流程架构图设计 (如适用)
- **图表语法规范**：如任务本身含有复杂的底层时序调用、状态机或逻辑分支，必须使用 Markdown 的 `mermaid` 代码块编写（如 `sequenceDiagram` 或 `stateDiagram`），不要直接截图，以便后续代码与逻辑追踪。
