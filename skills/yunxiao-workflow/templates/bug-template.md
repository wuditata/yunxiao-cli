# BUG规范 (Bug Template)

本模板适用于提交系统异常、线上/线下故障或代码缺陷。

## 1. 标题规范与字数限制
> **BUG标题规范**：`[环境/模块] 发生场景与现象表现`。示例：`[线上/支付页] 并发退款时报空指针异常`
> **字数限制**：正文控制在 300 字以内。尽可能用贴图、日志代码块代替繁冗的文字描述。

## 2. 环境信息
- **环境名称**：日常(Daily) / 预发(Pre) / 线上(Prod)
- **应用/服务**：涉及到的微服务或应用名称
- **版本/分支**：v1.2.0 / release-xxx
- **测试账号/前置条件**：提供重现时所需要登录或者设置好的账号与数据。

## 3. 详细现象 (实际结果)
描述当前系统的不符合预期的行为表现：
- 错误表现的具体描述。
- **关联日志/截图**：（如果是接口报错，提供对应的报错链路TraceId、HTTP返回头、以及控制台/服务端的Error堆栈等）。

## 4. 期望结果
系统原设定的正确运行应该是什么样的？
- 对比说明正确的结果应该显示什么，或返回什么值。

## 5. 复现步骤 (重中之重)
清晰、保姆级地列出重现步骤，以确保修复人员和AI能在本地完全再现该问题：
1. 登录该账号前往 `XXX` 页面。
2. 点击 `XXXX` 按钮，选择特定的条件。
3. 提交表单时拦截请求，观察异常信息。
4. 或者直接调用以下接口：`...`，重放请求。

## 6. (可选) 原因排查建议 / 排查记录
如果是开发或测试提交，如果已经有一些初步排查的结论，可一并提供：
- 初步判断在 `XXXX` 服务处理异步回调的地方，因为未判空导致。
