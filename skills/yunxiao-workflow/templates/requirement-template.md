# 需求规范 (Requirement Template)

本模板适用于描述业务线的新功能需求。为了让团队成员和AI开发助手更准确地理解并执行，建议使用如下格式：

## 1. 标题规范与字数限制
> **需求标题规范**：`[模块/归属] 需求简述`。示例：`[用户运营] 新增用户画像标签`
> **字数限制**：正文控制在 300-500 字以内。要求言简意赅，不要长篇大论。

## 2. 价值与背景 (Why)
- **痛点 / 问题解决**：当前系统缺失了什么？用户有什么抱怨？
- **业务价值**：提升了哪些指标（例如转化率、效率等）？
- **目标用户**：此功能面向哪部分使用者群体？

## 3. 核心功能描述 (What)
列出需求需要实现的具体功能点：
1. **功能点 1**：具体描述。
2. **功能点 2**：具体描述。

## 4. 界面/交互 (可选)
- 有无涉及前端 UI 改动？
- 相关的原型链接/设计稿链接：[链接地址]

## 5. 验收标准 / DoD (Done of Definition)
> 清晰可验证的结果标准，开发和AI依据此进行自测与验证。
- [ ] 前端：完成了XX页面的渲染，且无报错。
- [ ] 后端：提供的XX接口通过了所有的参数校验和返回字段验证。
- [ ] 整体测试流转：完成了XX场景下的正向和逆向用例（如边界值异常等）。

## 6. 其他 / 补充信息
- **流程/架构图规范**：若业务涉及复杂逻辑流转，需使用 Markdown 护航的 `mermaid` 语法进行说明（如 `graph TD` 或 `sequenceDiagram`），保障 AI 与多平台渲染兼容。
- 前端性能或扩展要求？
- 外部接口依赖与调用频次？
- 前置或并行的其他业务需求？
